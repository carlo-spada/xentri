# Product Soul Validation Checklist

> This checklist is used by the validate-product-soul workflow to assess document quality.

## Document Structure

- [ ] Frontmatter present (Date, Author, Context)
- [ ] No placeholder text remains (`{{variable}}`, `[TODO]`, `[NEEDS CONFIRMATION]`)
- [ ] Markdown headers properly structured
- [ ] Sections logically ordered

## Executive Summary

- [ ] Product concept explained in 1-2 clear sentences
- [ ] Primary problem clearly identified
- [ ] Target market specifically named (not generic)
- [ ] Value proposition compelling and differentiated
- [ ] Summary accurately reflects full document content

## Problem Statement

- [ ] Pain points are specific and concrete
- [ ] Impact quantified where possible (time, money, opportunities)
- [ ] Explanation of why existing solutions fall short
- [ ] Urgency for solving now is justified
- [ ] Problem validated with evidence or reasoning

## Solution Definition

- [ ] Core approach clearly explained
- [ ] No premature implementation details
- [ ] Key differentiators from existing solutions identified
- [ ] Solution aligns directly with stated problems
- [ ] Vision paints clear picture of user experience

## Target Users

- [ ] Primary user segment has specific profile
- [ ] Avoids generic personas ("busy professionals", "tech-savvy users")
- [ ] User behaviors and workflows documented
- [ ] Specific pain points tied to user segments
- [ ] User goals clearly articulated

## MVP Scope

- [ ] Core features list contains only true must-haves
- [ ] Each core feature has rationale for inclusion
- [ ] Out of scope section explicitly lists deferred features
- [ ] Scope is genuinely minimal and viable
- [ ] No feature creep evident in "must-have" list

## Optional Sections (if present)

### Success Metrics

- [ ] Metrics are measurable with clear targets
- [ ] Metrics align with problem statement
- [ ] KPIs follow SMART criteria

### Market Context

- [ ] Competitive landscape understood
- [ ] Market opportunity reasonably sized
- [ ] Differentiation strategy clear

### Technical Preferences

- [ ] Platform choices specified
- [ ] Marked as preferences, not decisions
- [ ] Integration requirements identified

### Risks and Assumptions

- [ ] Key risks identified with potential impact
- [ ] Assumptions are testable
- [ ] Open questions are specific and answerable

## Overall Quality

- [ ] Language is clear and jargon-free
- [ ] Terminology used consistently throughout
- [ ] Internal consistency (no contradictions)
- [ ] Problem → Solution → Users → Features flow logically

## Readiness Assessment

### Critical Issues (blocks downstream work):

- [ ] None identified

### Warnings (should address):

- [ ] None identified

### Ready for PRD Creation:

- [ ] Yes, Product Soul is validated
- [ ] No, requires amendments (see issues above)
